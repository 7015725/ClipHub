# ClipHub 模块规范

## 1. 结构结论

ClipHub 使用单个 GitHub 仓库、一个 ShortX 入口文件和多个 Rhino ES5 JavaScript 子模块。这里的“子模块”是项目内部文件，不是 Git Submodule。

用户只需要在 ShortX 的 JavaScript 任务中粘贴并执行完整 `ClipHub.js`。运行模块由入口自动下载、校验、缓存和加载，不要求用户通过 Termux 手动复制 `src/`。

## 2. 源码与运行目录

源码目录：

```text
ClipHub/
├── ClipHub.js
├── module-manifest.json
├── src/
└── scripts/
```

ShortX 运行目录：

```text
shortx.getShortXDir()/ClipHub/
├── modules/
│   ├── ch_01_base.js
│   └── ...
├── data/
├── logs/
├── cache/
│   └── module-manifest.local.json
└── probes/
```

仓库中的 `src/ch_01_base.js` 对应运行目录中的 `modules/ch_01_base.js`。数据库、日志、缓存和下载后的模块副本不得提交到仓库。

`ClipHub.js` 本身保存在 ShortX 任务中，不依赖运行目录中存在另一个入口副本。

## 3. 模块下载流程

入口通过公开的 `raw.githubusercontent.com` 文件地址读取 manifest 和模块，不使用匿名 GitHub REST Contents API，因此不受 REST API 每小时低额度限制。

入口启动顺序：

```text
读取远程 module-manifest.json
→ 校验 manifest 结构和入口最低版本
→ 校验本地 15 个模块的 Git blob SHA
→ 无变化时直接加载本地模块
→ 首次运行或版本变化时下载到 modules.stage
→ 校验每个下载文件的 Git blob SHA
→ 当前 modules 重命名为 modules.backup
→ modules.stage 原子切换为 modules
→ 加载全部模块并启动 App
→ 启动成功后删除 modules.backup
```

如果新模块下载、校验、加载或启动失败：

- 不执行不完整的临时目录；
- 恢复上一套 `modules`；
- 恢复上一份本地 manifest；
- 上一套模块完整时允许离线回退运行；
- 首次运行且网络不可用时明确返回失败，不生成半套模块。

当前开发阶段默认远程 ref：

```text
agent/initialize-project-skeleton
```

合并正式版本前必须切换为：

```text
main
```

## 4. Manifest 约束

`module-manifest.json` 至少包含：

```text
schemaVersion
moduleSetVersion
entryMinVersion
sourceRef
modules[].name
modules[].path
modules[].sha
```

其中 `modules[].sha` 是 Git blob SHA-1。raw 文件响应不提供 blob SHA，因此入口对下载后的 UTF-8 内容重新计算 Git blob SHA；只有与 manifest 完全一致时才写入运行目录。

修改任何 `src/ch_*.js` 后必须同步更新：

- 对应模块 SHA；
- `moduleSetVersion`。

后续应通过 GitHub Actions 自动生成 manifest，避免手工遗漏。

## 5. 模块加载顺序

入口按文件编号顺序加载 `ch_01_base.js` 至 `ch_15_app.js`。基础模块必须先于依赖模块加载，模块不得自行改变加载顺序。

## 6. 命名空间

所有模块必须挂载到 `ClipHub.<ModuleName>`，禁止暴露 `openWindow`、`saveItem`、`closeDatabase` 等通用全局函数。

普通模块至少提供：

```text
MODULE_NAME
MODULE_VERSION
init(context)
shutdown()
```

`init` 失败必须抛出错误；`shutdown` 必须幂等，并允许在部分初始化状态下调用。

## 7. 主要边界

- `Database` 管理 SQLite 连接、结构和迁移。
- `Repository` 集中处理数据访问，UI 不直接写 SQL。
- `Clipboard` 管理监听、去重和自身回写抑制。
- `Classifier` 只返回识别结果，不直接修改 UI 或数据库。
- `EventBus` 隔离模块依赖。
- `Window` 管理主 WindowManager 生命周期和主窗口布局状态。
- `List` 管理主列表与详情窗口的显示状态，并提供当前最上层页面的关闭能力。
- `Editor` 管理新增、编辑和标签窗口，并负责未保存内容的返回边界。
- `Filter` 管理筛选窗口，并提供幂等关闭能力。
- `Navigation` 管理系统返回注册、预测性返回视觉、窗口焦点、前后台判断和统一 UI 隐藏链。
- `App` 负责初始化、异常回滚、反向清理、单实例锁和正式控制入口。
- 入口只负责远程同步、完整性校验、模块加载和启动，不实现业务功能。

### 7.1 当前 Navigation 承载位置

当前入口版本固定加载 15 个模块。为避免在阶段 2D 同时修改入口模块数量、加载顺序和清单结构，`ClipHub.Navigation` 暂时承载在：

```text
src/ch_12_translation.js
```

该文件同时继续导出 `ClipHub.Translation`。两者具有独立命名空间、版本号和关闭接口，不共享翻译配置或业务状态。

这是阶段性边界，不代表系统手势属于翻译功能。后续入口版本允许调整模块数量时，应将其拆分为独立的 `ch_16_navigation.js`，并同步更新入口、Manifest 和模块规范。

## 8. 系统手势模块边界

系统手势与 UI 退出是首版必要功能，不允许由单个页面临时实现。

### 8.1 返回分发

`Navigation` 统一接收：

- Android 系统侧滑返回；
- `OnBackInvokedDispatcher`；
- Android 14 预测性返回进度、取消和提交；
- 三键返回键和 Escape 后备事件；
- 后续页面内返回按钮调用。

返回分发器只能判断当前 UI 层级并调用对应模块的关闭接口，不直接操作数据库、剪贴板或业务数据。

固定关闭顺序：

```text
Filter / Editor / Detail / Tag
              ↓
主列表首页
              ↓
Navigation.hideUi()
```

首页返回必须调用统一 UI 隐藏链，不能调用 `App.stop()`。

### 8.2 窗口发现与注册

当前各 UI 页面分别创建 WindowManager View，因此 `Navigation` 通过进程内 WindowManager View 集合发现标题以 `ClipHub` 开头的窗口，并在页面显示后注册返回回调。

要求：

- 仅处理 ClipHub 自己的窗口；
- 同一个 View 只注册一次；
- View 分离时注销返回回调和焦点监听；
- Android 14 优先使用 `OnBackAnimationCallback`；
- Android 13 回退到 `OnBackInvokedCallback`；
- 更低版本或回调不可用时保留按键后备链；
- 主窗口必须保持 `FLAG_NOT_TOUCH_MODAL`，不得阻断窗口外部应用操作。

### 8.3 进入最近任务或后台

前后台状态由 `Navigation` 集中判断，页面模块不得各自根据焦点变化独立关闭整个应用。

判断信号包括：

- ClipHub 窗口失去焦点；
- 聚焦 Root Task 的 activity type；
- 前台包名切换；
- Launcher、SystemUI、Quickstep 或最近任务界面成为前台。

判断采用延迟复核，避免次级窗口替换主窗口时把正常页面切换误判为进入后台。

`Navigation.hideUi(reason)` 按以下顺序幂等关闭：

1. `Filter.closePanel()`；
2. `Editor.close()`；
3. `List.closeDetail()`；
4. 其他次级窗口关闭接口；
5. `List.hide(true)` 或 `Window.close()`；
6. 清除返回注册、遮暗层、临时触摸层和窗口状态。

该流程不得停止：

- `Clipboard` 监听；
- `Database`；
- `EventBus`；
- 运行锁；
- 动态控制入口。

### 8.4 幂等要求

所有 UI 模块的关闭接口必须：

- 未打开时返回成功或“已关闭”状态；
- 不重复移除同一个 View；
- 在 `removeViewImmediate` 异常时核对实际附着状态；
- 无论成功或失败都清理模块内 View 引用；
- 记录关闭原因但不记录剪贴板正文；
- 允许系统返回、后台切换和控制命令在短时间内连续触发。

`Navigation` 额外使用短时间重复返回抑制，防止预测性返回最终回调和按键后备链在设备定制系统上重复提交。

### 8.5 禁止项

- 禁止覆盖屏幕左右边缘的透明 View 模拟侧滑返回；
- 禁止覆盖底部系统导航区域拦截上拉手势；
- 禁止系统进入后台时自动保存编辑内容；
- 禁止系统返回直接停止整个 ClipHub 实例；
- 禁止各页面维护互相冲突的返回栈；
- 禁止关闭后遗留全屏透明遮罩。

## 9. Rhino ES5 约束

禁止使用 `let`、`const`、`class`、箭头函数、模板字符串、可选链、空值合并和 `for...of`。

检查命令：

```sh
python3 scripts/validate_es5.py .
```

静态检查不能替代 Android 14 / ShortX 真机测试。

## 10. 当前更新边界

当前已实现：

- raw GitHub 文件下载；
- manifest 版本检查；
- Git blob SHA 校验；
- 临时目录下载；
- 旧模块备份与失败恢复；
- 完整本地缓存的离线回退；
- `ClipHub.Navigation` 统一返回分发；
- Android 13 `OnBackInvokedCallback`；
- Android 14 `OnBackAnimationCallback`；
- 次级页面到首页的返回层级；
- 首页返回关闭全部 UI；
- 最近任务和后台切换后的 UI 隐藏判断；
- UI 隐藏后后台监听继续运行；
- 系统手势专项探测 030。

当前待验证或暂未实现：

- 探测 030 真机结果；
- 真实侧滑返回、三键返回和底部上拉人工复核；
- 平板与多窗口下的系统手势复核；
- 将 `Navigation` 拆分为独立模块；
- RSA/Ed25519 独立签名；
- Stable/Beta UI 通道切换；
- 下载镜像源切换；
- 差量下载；
- 自动更新入口文件本身。

入口文件变更时仍需要用户手动替换 ShortX 任务中的完整 `ClipHub.js`。
