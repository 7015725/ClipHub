# ClipHub 模块规范

## 1. 结构结论

ClipHub 使用单个 GitHub 仓库、一个 ShortX 入口文件和多个 Rhino ES5 JavaScript 子模块。这里的“子模块”是项目内部文件，不是 Git Submodule。

用户只需要在 ShortX 的 JavaScript 任务中粘贴并执行完整 `ClipHub.js`。运行模块由入口自动下载、校验、缓存和加载，不要求用户通过 Termux 手动复制 `src/`。

## 2. 源码与运行目录

源码目录：

```text
ClipHub/
├── ClipHub.js
├── module-manifest.json
├── src/
└── scripts/
```

ShortX 运行目录：

```text
shortx.getShortXDir()/ClipHub/
├── modules/
│   ├── ch_01_base.js
│   └── ...
├── data/
├── logs/
├── cache/
│   └── module-manifest.local.json
└── probes/
```

仓库中的 `src/ch_01_base.js` 对应运行目录中的 `modules/ch_01_base.js`。数据库、日志、缓存和下载后的模块副本不得提交到仓库。

`ClipHub.js` 本身保存在 ShortX 任务中，不依赖运行目录中存在另一个入口副本。

## 3. 模块下载流程

入口通过公开的 `raw.githubusercontent.com` 文件地址读取 manifest 和模块，不使用匿名 GitHub REST Contents API，因此不受 REST API 每小时低额度限制。

入口启动顺序：

```text
读取远程 module-manifest.json
→ 校验 manifest 结构和入口最低版本
→ 校验本地 15 个模块的 Git blob SHA
→ 无变化时直接加载本地模块
→ 首次运行或版本变化时下载到 modules.stage
→ 校验每个下载文件的 Git blob SHA
→ 当前 modules 重命名为 modules.backup
→ modules.stage 原子切换为 modules
→ 加载全部模块并启动 App
→ 启动成功后删除 modules.backup
```

如果新模块下载、校验、加载或启动失败：

- 不执行不完整的临时目录；
- 恢复上一套 `modules`；
- 恢复上一份本地 manifest；
- 上一套模块完整时允许离线回退运行；
- 首次运行且网络不可用时明确返回失败，不生成半套模块。

当前开发阶段默认远程 ref：

```text
agent/initialize-project-skeleton
```

合并正式版本前必须切换为：

```text
main
```

## 4. Manifest 约束

`module-manifest.json` 至少包含：

```text
schemaVersion
moduleSetVersion
entryMinVersion
sourceRef
modules[].name
modules[].path
modules[].sha
```

其中 `modules[].sha` 是 Git blob SHA-1。raw 文件响应不提供 blob SHA，因此入口对下载后的 UTF-8 内容重新计算 Git blob SHA；只有与 manifest 完全一致时才写入运行目录。

修改任何 `src/ch_*.js` 后必须同步更新：

- 对应模块 SHA；
- `moduleSetVersion`。

后续应通过 GitHub Actions 自动生成 manifest，避免手工遗漏。

## 5. 模块加载顺序

入口按文件编号顺序加载 `ch_01_base.js` 至 `ch_15_app.js`。基础模块必须先于依赖模块加载，模块不得自行改变加载顺序。

## 6. 命名空间

所有模块必须挂载到 `ClipHub.<ModuleName>`，禁止暴露 `openWindow`、`saveItem`、`closeDatabase` 等通用全局函数。

普通模块至少提供：

```text
MODULE_NAME
MODULE_VERSION
init(context)
shutdown()
```

`init` 失败必须抛出错误；`shutdown` 必须幂等，并允许在部分初始化状态下调用。

## 7. 主要边界

- `Database` 管理 SQLite 连接、结构和迁移。
- `Repository` 集中处理数据访问，UI 不直接写 SQL。
- `Clipboard` 管理监听、去重和自身回写抑制。
- `Classifier` 只返回识别结果，不直接修改 UI 或数据库。
- `EventBus` 隔离模块依赖。
- `Window` 只管理 WindowManager 生命周期。
- `App` 负责初始化、异常回滚、反向清理和单实例锁。
- 入口只负责远程同步、完整性校验、模块加载和启动，不实现业务功能。

## 8. Rhino ES5 约束

禁止使用 `let`、`const`、`class`、箭头函数、模板字符串、可选链、空值合并和 `for...of`。

检查命令：

```sh
python3 scripts/validate_es5.py .
```

静态检查不能替代 Android 14 / ShortX 真机测试。

## 9. 当前更新边界

当前已实现：

- raw GitHub 文件下载；
- manifest 版本检查；
- Git blob SHA 校验；
- 临时目录下载；
- 旧模块备份与失败恢复；
- 完整本地缓存的离线回退。

当前暂未实现：

- RSA/Ed25519 独立签名；
- Stable/Beta UI 通道切换；
- 下载镜像源切换；
- 差量下载；
- 自动更新入口文件本身。

入口文件变更时仍需要用户手动替换 ShortX 任务中的完整 `ClipHub.js`。
