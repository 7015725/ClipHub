# ClipHub 最近任务自动关闭探测 032

## 目标

验证模块集 `20260722.25` 在 ColorOS / Android 14 中不依赖 WindowManager 焦点丢失回调，也能在用户从底部上拉进入最近任务后关闭全部 ClipHub UI。

## 探测 031 根因

探测 031 已确认：

```text
baselinePackage=tornaco.apps.shortx
进入最近任务后 packageName=com.android.launcher
activityType=2
ClipHub Root.hasWindowFocus=true
focusLossCount=0
backgroundCheckCount=0
```

因此 `.24` 的失败原因是 Overlay Root 在最近任务界面仍保持 `hasWindowFocus=true`，焦点丢失回调没有启动后台检查。

## 模块集 20260722.25

新增 `ClipHub.RecentsWatch`，暂时承载于 `ch_14_event_bus.js`：

- 仅在 ClipHub UI 可见时运行；
- 每 180ms 读取一次聚焦 Root Task；
- 连续两个采样确认前台包、Activity Type 或 Task ID 已离开基线；
- 不依赖 Overlay Window 的 `hasWindowFocus()`；
- 命中后调用既有 `ClipHub.Navigation.hideUi()`；
- UI 关闭后立即停止采样；
- 不停止剪贴板监听、SQLite、运行锁或控制入口；
- 不覆盖底部导航区，不拦截任何系统手势。

为避免在桌面直接打开 ClipHub 时立即关闭，观察器使用打开时的包名、Activity Type 和 Task ID 作为基线。仅在后续任务状态相对基线发生变化时隐藏 UI。

Android 14 / ColorOS 真机结果：

```text

```

观察器初始基线：

```text

```

底部上拉进入最近任务后：

```text
lastPackage=com.android.launcher
lastActivityType=2
lastTaskId=1
lastSignalReason=top_package_changed
confirmedSignalCount=1
hideCount=1
lastHideReason=task_watch_top_package_changed
```

统一关闭链结果：

```text
uiClosedAfterRealRecents=true
callbacksCleared=true
registeredRootCount=0
navigationHideTriggered=true
appStillRunning=true
lockReleased=true
lastError=null
```

观察器在第一次变化样本后完成二次确认并关闭 UI：

```text
elapsedMs=2075：检测到第一个前台任务变化信号
elapsedMs=2199：UI 已关闭，返回回调注册项已清零
```

`backgroundCheckCount=0` 和 `backgroundHideCount=0` 属于预期结果，因为本次命中的是新的 Root Task 观察器，而不是旧的焦点丢失后备链。

自动验证结论：模块集 `.25` 已在真实最近任务切换中成功关闭 ClipHub UI，不依赖 ColorOS 未触发的 Overlay 焦点丢失事件。

## 运行步骤

1. 执行 `probes/cliphub_stop_formal.js`；
2. 运行 `ClipHub.js`，安装模块集 `.25`；
3. 再次停止正式实例；
4. 将 `probes/cliphub_recent_tasks_auto_hide_probe_032.js` 完整复制到 ShortX 新任务；
5. 运行探测 032；
6. 窗口出现后 3 秒内上拉进入最近任务；
7. 停留至少 3 秒，等待探测约 12 秒自动结束。
