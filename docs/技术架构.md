# ClipHub 技术架构

## 1. 架构目标

ClipHub 需要在 ShortX 的 Rhino ES5 环境中稳定运行，同时管理以下状态：

- 系统剪贴板监听；
- SQLite 数据；
- WindowManager 悬浮窗口；
- 搜索与过滤；
- 内容识别；
- 编辑和标签；
- 翻译扩展；
- 生命周期与异常恢复。

设计上应避免把监听、数据库和 View 逻辑集中在单个文件中。

## 2. 建议分层

```text
入口层
  └── ClipHub.js

基础设施层
  ├── 日志
  ├── 配置
  ├── 数据库
  ├── 线程调度
  └── WindowManager 生命周期

领域层
  ├── 剪贴板记录
  ├── 内容分类
  ├── 标签
  ├── 搜索过滤
  └── 去重与回环抑制

界面层
  ├── 主悬浮窗
  ├── 列表
  ├── 编辑器
  ├── 筛选器
  ├── 标签管理
  └── 设置页
```

## 4. 剪贴板数据流

```text
ClipboardManager 变化
        ↓
读取 ClipData
        ↓
转换为安全字符串
        ↓
自身写入检查
        ↓
规范化与摘要计算
        ↓
短时间重复判断
        ↓
来源应用探测
        ↓
内容类型识别
        ↓
SQLite 事务保存
        ↓
通知已打开的 UI 增量刷新
```

## 5. 回环抑制

ClipHub 将历史项目写回剪贴板时，监听器会再次收到变化。

建议同时使用：

1. 最近一次主动写入文本摘要；
2. 主动写入时间戳；
3. 短时间抑制窗口；
4. 数据库最近记录摘要；
5. 监听事件序号。

不能只依赖固定延时，因为不同设备的剪贴板事件时序可能不同。

建议状态：

```text
lastWriteHash
lastWriteAt
lastObservedHash
lastObservedAt
clipboardEventSeq
```

## 6. 去重策略

需要区分：

- 完全相同文本；
- 仅首尾空白不同；
- 多次复制相同内容但来源不同；
- 用户希望保留重复历史。

首版建议：

- 原始文本完整保存；
- 生成 `normalized_hash` 用于快速比较；
- 默认在短时间窗口内合并完全相同记录；
- 合并时更新最后复制时间和复制次数；
- 不跨越较长时间永久去重。

## 7. SQLite 初步结构

### 7.1 `clipboard_items`

```sql
CREATE TABLE clipboard_items (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    content TEXT NOT NULL,
    normalized_hash TEXT NOT NULL,
    content_type TEXT NOT NULL DEFAULT 'text',
    source_package TEXT,
    source_label TEXT,
    source_uid INTEGER,
    source_confidence INTEGER NOT NULL DEFAULT 0,
    is_pinned INTEGER NOT NULL DEFAULT 0,
    manual_order INTEGER NOT NULL DEFAULT 0,
    copy_count INTEGER NOT NULL DEFAULT 1,
    created_at INTEGER NOT NULL,
    last_copied_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL,
    deleted_at INTEGER
);
```

建议索引：

```sql
CREATE INDEX idx_clipboard_items_last_copied
ON clipboard_items(last_copied_at DESC);

CREATE INDEX idx_clipboard_items_type
ON clipboard_items(content_type);

CREATE INDEX idx_clipboard_items_source
ON clipboard_items(source_package);

CREATE INDEX idx_clipboard_items_hash
ON clipboard_items(normalized_hash);
```

### 7.2 `tags`

```sql
CREATE TABLE tags (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    normalized_name TEXT NOT NULL UNIQUE,
    color_value INTEGER,
    manual_order INTEGER NOT NULL DEFAULT 0,
    created_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL
);
```

### 7.3 `clipboard_item_tags`

```sql
CREATE TABLE clipboard_item_tags (
    item_id INTEGER NOT NULL,
    tag_id INTEGER NOT NULL,
    created_at INTEGER NOT NULL,
    PRIMARY KEY (item_id, tag_id)
);
```

### 7.4 `settings`

```sql
CREATE TABLE settings (
    key TEXT PRIMARY KEY,
    value TEXT,
    updated_at INTEGER NOT NULL
);
```

### 7.5 `schema_meta`

```sql
CREATE TABLE schema_meta (
    key TEXT PRIMARY KEY,
    value TEXT
);
```

## 8. 排序规则

显示顺序建议拆为：

1. 置顶记录；
2. 用户手动排序；
3. 最后复制时间。

不要直接重写创建时间实现拖动排序。

可采用稀疏排序值，例如：

```text
1000, 2000, 3000...
```

在两项之间插入时取中间值；空间不足时再统一重排。

## 9. 搜索方案

首版可使用 SQL `LIKE`：

```sql
WHERE content LIKE ?
   OR source_label LIKE ?
   OR source_package LIKE ?
```

注意：

- 参数必须绑定，禁止拼接用户输入；
- 对超长文本应限制列表预览长度；
- 搜索结果仍返回完整记录 ID；
- 后续数据量较大时再评估 FTS。

## 10. WindowManager 边界

必须统一管理：

```text
windowAdded
windowToken
windowParams
rootView
ownerHandler
isClosing
isCollapsed
isPinned
```

要求：

- `addView`、`updateViewLayout`、`removeViewImmediate` 在所属线程执行；
- 所有关闭路径进入同一个幂等清理函数；
- 添加失败时清理半初始化状态；
- 不重复添加同一个 View；
- 屏幕尺寸变化后重新计算安全区域；
- 异常退出时移除透明遮罩；
- 拖动过程中只更新必要的位置参数。

## 11. 拖动冲突处理

需要三种独立手势：

1. 标题栏拖动整个窗口；
2. 项目左侧手柄拖动排序；
3. 列表本身垂直滚动。

边界：

- 标题栏按钮区域不得启动窗口拖动；
- 排序只能从明确手柄启动；
- 达到移动阈值前不抢占点击；
- 横向或纵向意图判断不能误伤列表滚动；
- `ACTION_CANCEL` 必须恢复临时状态。

## 12. UI 更新策略

不建议每次剪贴板变化都重建整个悬浮窗。

建议：

- 数据层发出“新增、更新、删除、重排”事件；
- UI 对可见项目执行局部更新；
- 搜索状态下重新执行当前查询；
- 动画期间避免重复提交相同状态；
- 无数据变化时不触发重绘或动画。

## 13. 日志边界

日志允许记录：

- 事件类型；
- 内容长度；
- 内容摘要前缀或哈希；
- 来源包名；
- 数据库耗时；
- UI 生命周期；
- 线程 ID；
- 错误类型。

日志默认禁止记录：

- 完整剪贴板正文；
- 翻译密钥；
- 访问令牌；
- 数据库导出内容；
- 疑似密码和验证码。

## 14. 异常恢复

至少覆盖：

- 数据库打开失败；
- 数据库迁移失败；
- ClipboardManager 不可用；
- WindowManager 添加失败；
- UI 线程已结束；
- 来源应用探测失败；
- 翻译网络失败；
- 内容过长；
- 非文本 ClipData；
- 脚本重复启动。

## 15. 与 ToolHub 的边界

ClipHub 应作为独立项目运行：

- 独立入口；
- 独立数据库；
- 独立日志目录；
- 独立窗口状态；
- 独立更新流程。

未来可以提供 ToolHub 按钮或广播入口，但 ToolHub 不应直接操作 ClipHub 内部数据库结构。
