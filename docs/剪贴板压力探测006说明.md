# ClipHub 剪贴板压力探测 006

## 1. 目的

`cliphub_pressure_probe_006.js` 同时验证：

1. 正式 ClipHub 实例的跨 ShortX 任务停止控制；
2. 随机控制端点、确认文件和正式文件锁释放；
3. 100 条不同文本连续复制；
4. 多 item 文本读取；
5. 空白、非文本和超长 ClipData 忽略；
6. 停止清理、剪贴板恢复和正式实例重启。

## 2. 当前版本

```text
moduleSetVersion=20260721.7
probeVersion=2
```

## 3. 前置条件

必须满足：

- 设备已释放旧模块集实例；
- ShortX 入口已同步到模块集 `20260721.7` 或更高；
- 正式 ClipHub 当前正在运行；
- 入口返回中存在：

```text
controlTransport=dynamic_broadcast_token
controlEndpointPath=<ShortX 私有目录>/ClipHub/cache/control_endpoint.json
```

- 本地运行清单：

```text
shortx.getShortXDir()/ClipHub/cache/module-manifest.local.json
```

其中 `moduleSetVersion` 必须为 `20260721.7` 或更高。

006 会严格要求正式实例正在持有：

```text
shortx.getShortXDir()/ClipHub/data/cliphub.lock
```

如果正式实例未运行或控制端点不存在，006 会直接失败，不执行压力测试，也不会继续修改剪贴板。

### 新探测脚本与旧运行模块混用

仅更新 `cliphub_pressure_probe_006.js` 并不等于更新设备运行目录中的 15 个模块。

如果结果出现：

```text
formalInitiallyRunning=false
formalEndpointPresent=false
```

并且 `formalRestart` 返回中缺少：

```text
controlTransport
controlEndpointPath
```

说明设备上的运行模块仍是旧模块集。需要重启设备后先执行完整 `ClipHub.js` 入口完成模块同步，再运行 006。

## 4. 控制端点

正式实例每次启动生成新的 192-bit 随机令牌，并在 ShortX 私有目录写入：

```text
shortx.getShortXDir()/ClipHub/cache/control_endpoint.json
```

端点包含：

```text
schemaVersion=1
transport=dynamic_broadcast_token
action=<固定前缀 + 随机令牌>
token=<随机令牌>
runtimeDir=<正式运行目录>
```

Android 13+ 的动态接收器使用 `RECEIVER_EXPORTED`，但只有同时匹配以下条件才执行停止：

- 随机 action；
- 随机令牌；
- 精确 runtimeDir；
- command=stop。

端点文件位于 ShortX 私有目录。探测结果和日志不会输出随机令牌。

停止完成后，正式实例会删除端点文件，并写入 requestId 对应确认文件。

## 5. 隔离边界

压力测试使用：

```text
shortx.getShortXDir()/ClipHubProbe006/
```

不会把 100 条压力测试文本写入正式：

```text
shortx.getShortXDir()/ClipHub/data/cliphub.db
```

测试完成后会：

- 停止隔离实例；
- 关闭隔离数据库；
- 恢复执行前的原剪贴板；
- 重新启动正式 ClipHub；
- 删除 `ClipHubProbe006` 隔离目录。

报告不包含原剪贴板正文。

## 6. 执行方法


```sh

```


## 7. 测试内容

### 7.1 跨任务停止控制

006 从私有端点读取本次随机 action 和令牌，发送：

```text
command=stop
runtimeDir=<正式 ClipHub 运行目录>
requestId=<本次探测 ID>
controlToken=<私有随机令牌>
```

正式实例应：

- 在 `main` 线程接收；
- 校验 action、令牌和运行目录；
- 注销控制接收器；
- 删除控制端点文件；
- 注销 ClipboardManager 监听器；
- 关闭数据库；
- 释放文件锁；
- 写入 requestId 对应确认文件。

### 7.2 100 条不同文本

006 以约 25 ms 间隔写入 100 条不同文本，要求隔离数据库最终准确保存 100 条记录，且无错误事件。

### 7.3 ClipData 边界

随后验证：

- 两个文本 item 合并为带换行的一条记录；
- 纯空白文本被忽略；
- `text/uri-list` 非文本内容被忽略；
- 超过探测配置 `maxChars=4096` 的文本被忽略；
- 无效 ClipData 不增加数据库记录。

## 10. 失败处理

- `formalInitiallyRunning=false`：设备重启后未先执行正式入口，或正式实例已被停止；
- `formalEndpointPresent=false`：当前正式实例不是模块集 `20260721.7+`，或端点初始化失败；
- `formalAckReceived=false`：检查随机 action 投递、令牌校验和接收器存活状态；
- `formalEndpointRemoved=false`：检查停止清理顺序；
- `formalAckThreadName!=main`：控制接收器线程边界异常；
- `uniqueCount<100`：检查监听回调丢失、数据库写入错误或等待时间；
- `uniqueCount>100`：检查同一 clip 重复回调抑制；
- `invalidCountStable=false`：检查空白、非文本或超长内容过滤；
- `formalRestart` 失败：检查文件锁、接收器注销和数据库关闭顺序。

不得删除仍被正式实例持有的 `cliphub.lock` 文件。
