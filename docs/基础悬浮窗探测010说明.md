# ClipHub 基础悬浮窗探测 010

## 1. 目的

`cliphub_window_probe_010.js` 用于验证阶段 2A 的最小 WindowManager 悬浮窗骨架：

- `TYPE_APPLICATION_OVERLAY` 创建；
- 所有 add/update/remove 操作切换到 Android `main` 线程；
- View 真实完成 `isAttachedToWindow()`；
- 重复打开复用同一窗口；
- 状态栏、导航栏和刘海安全区域计算；
- 左上、右下边界夹紧；
- 标题栏拖动监听器安装；
- 折叠、展开、固定状态；
- 幂等关闭；
- 关闭后重新打开；
- App 停止时完整释放 WindowManager View。

## 2. 当前版本

```text
moduleSetVersion=20260721.10
probeVersion=2
```

v2 不修改运行模块，只修正 v1 的首帧附着时序误判。

## 3. 附着等待

`WindowManager.addView()` 返回表示窗口已被接受，但当前 ShortX/system_server 环境中，View 的首帧 attach 可能稍后才由主线程完成。

v2 在首次打开和二次打开后分别：

- 最多等待 1500 ms；
- 每 25 ms 检查一次 `attached && attachedToWindow`；
- 观察到真实附着后才继续移动、折叠和关闭；
- 输出 `firstAttachWaitMs` 与 `reopenAttachWaitMs`；
- 真实附着后保留 100 ms 再移除。

如果 1500 ms 内仍未附着，探测会保持 `ok=false`，不会把 `addView()` 成功误当成可见窗口成功。

## 4. 实现边界

本阶段只实现 WindowManager 生命周期骨架，不实现：

- 剪贴板历史列表；
- 搜索、标签和编辑器；
- 窗口位置 SQLite 持久化；
- 屏幕旋转监听；
- 输入法和可聚焦编辑框；
- 完整视觉设计。

窗口默认使用不可聚焦、可触摸的 `TYPE_APPLICATION_OVERLAY`，不会抢占当前应用输入焦点。

## 5. 隔离边界

010 使用：

```text
shortx.getShortXDir()/ClipHubProbe010/
```

探测会先停止正式 ClipHub，再在隔离实例中两次创建和移除窗口，最后恢复正式实例。

不会修改正式数据库或系统剪贴板。

## 6. 执行顺序

设备当前已经安装模块集 `20260721.10` 时，只需要更新并执行 v2 探测脚本，不需要重新停止并同步模块。

```sh
cd ~/projects/github/ClipHub
git fetch origin
git switch agent/initialize-project-skeleton
git pull --ff-only

grep -n "probeVersion" probes/cliphub_window_probe_010.js
```

应看到：

```text
probeVersion: 2
```

复制：

```sh
cp probes/cliphub_window_probe_010.js \
  "/storage/emulated/0/Download/ClipHub_基础悬浮窗探测_010.js"
```

当前正式实例必须正在运行。010 会自行完成隔离停止和恢复。

## 8. 可见现象

探测执行期间会显示一个极简 ClipHub 窗口，并完成：

- 等待首帧真实附着；
- 左上角夹紧；
- 右下角夹紧；
- 折叠和展开；
- 短暂停留后关闭；
- 二次打开并再次等待附着；
- 最终关闭。

这属于预期探测行为。
