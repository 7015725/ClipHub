# ClipHub 真机探测说明

## 1. 目的

在实现 SQLite 和 ClipboardManager 监听前，先验证 ClipHub 项目骨架依赖的运行边界：

- ShortX 私有目录是否可读写；
- `eval` 加载模块时的变量作用域；
- IIFE 与统一命名空间是否能够隔离模块私有变量；
- `RandomAccessFile` / `FileChannel.tryLock()` 是否能够实现单实例保护；
- 文件锁释放后是否可以重新获取；
- 两个 ShortX 任务是否能够同时竞争同一个文件锁。

探测脚本不读取、不修改系统剪贴板。

## 2. 探测脚本

### `cliphub_runtime_probe_001.js`

自动执行：

1. 创建并检查以下目录：

```text
shortx.getShortXDir()/ClipHub/
├── modules/
├── data/
├── logs/
├── cache/
└── probes/
```

2. 测试 UTF-8 文件写入、读取、重命名和删除；
3. 测试当前入口采用的 direct `eval` 作用域；
4. 测试 IIFE 私有变量隔离；
5. 测试文件锁首次获取、持锁期间第二次获取被阻止、释放后重新获取；
6. 保存 JSON 报告。

报告路径：

```text
shortx.getShortXDir()/ClipHub/probes/
cliphub_runtime_probe_001_YYYYMMDD-HHMMSS.json
```

同时覆盖一份：

```text
cliphub_runtime_probe_001_latest.json
```

### `cliphub_lock_probe_002.js`

用于验证两个独立 ShortX 任务之间的并发文件锁。

操作步骤：

1. 在 ShortX 中建立两个独立 JS 任务 A、B；
2. 两个任务都粘贴 `cliphub_lock_probe_002.js` 的完整代码；
3. 运行 A；
4. 在 12 秒内运行 B；
5. A 应在约 12 秒后返回；
6. B 应快速返回被锁状态。

预期：

```json
任务 A：
{
  "ok": true,
  "role": "holder",
  "acquired": true,
  "blocked": false
}

任务 B：
{
  "ok": true,
  "role": "contender",
  "acquired": false,
  "blocked": true
}
```

如果 B 也成为 `holder`，说明：

- 两个任务没有共享同一 JVM 文件锁状态；
- 两个任务使用了不同的 ShortX 根目录；
- 首个任务已提前结束；
- 或当前环境的文件锁行为不适合作为唯一单实例策略。

## 3. 执行前检查

代码必须完整粘贴，不要只复制内部函数。

ShortX JS 任务应保持：

```text
JavaScript 引擎：Rhino
语法：ES5
单次任务：是
```

探测 002 会阻塞任务线程约 12 秒，仅用于一次性验证，不进入正式模块。

## 4. 结果判定

### 探测 001 通过

顶层结果：

```json
{
  "ok": true
}
```

并且以下字段均为 `true`：

```text
directory.ok
fileIo.ok
evalScope.ok
fileLock.ok
```

### `evalScope` 预期

```text
directVarVisibleInsideLoader = true
directVarLeakedToGlobal = false
namespaceVisibleGlobally = true
privateVarLeakedToGlobal = false
moduleEchoMatched = true
```

这表示：

- direct `eval` 声明的变量位于加载函数作用域；
- IIFE 模块能够挂载到全局 `ClipHub` 命名空间；
- IIFE 内部私有变量不会泄漏。

因此正式模块仍必须使用 IIFE，不能在模块文件顶层声明共享变量或函数。

### `fileLock` 预期

```text
firstAcquire = true
secondAttemptBlocked = true
secondAcquireWhileHeld = false
reacquireAfterRelease = true
```

## 5. 已知边界

探测 001 不能单独证明：

- 两个独立 ShortX 任务的并发锁竞争；
- system_server 被强制重启后的锁释放；
- Android 系统回收脚本上下文后的 Java 对象生命周期；
- 实际 `ClipboardManager` 监听线程；
- 实际 WindowManager 页面生命周期。

并发竞争由探测 002 补充。

## 6. 提交结果

执行后提供：

```text
cliphub_runtime_probe_001_latest.json
```

并发探测时再提供两份：

```text
cliphub_lock_probe_002_*.json
```

根据结果决定下一步：

- 全部通过：进入 SQLite schema 与迁移框架；
- `evalScope` 异常：调整模块加载器；
- 文件锁异常：改为文件锁 + 进程内全局状态双层保护；
- 目录读写异常：调整运行目录布局和权限策略。
