## 已确认的系统重启根因

真机崩溃包中，`system_server_watchdog` 和 `SYSTEM_SERVER_NATIVE_CRASH` 均无记录，`pstore` 为空；这是 Java/Rhino 主线程未捕获异常，不是 watchdog、native crash 或内核重启。

2026-07-23 06:28、06:36、06:53 三次 `system_server_crash` 均为同一异常：

```text
org.mozilla.javascript.EcmaError:
TypeError: Cannot call method "getResources" of null

script.displayMetrics(...:887)
→ script.readImeState(...:438)
→ delayed Runnable(...:806)
→ Handler.dispatchMessage
→ SystemServer.run
```

实际竞态：

```text
Editor 打开正文页
→ installEditorLayoutObserver() 投递 180 ms 延迟 IME 测量
→ 探测快速保存并执行 App.stop()
→ Editor.shutdown() 清空 appContext/windowManager/mainHandler
→ 延迟 Runnable 随后执行 readImeState()
→ displayMetrics() 的 WindowManager 路径失败
→ fallback 调用已为 null 的 appContext.getResources()
→ 未捕获异常终止 system_server
```

因此：

- 标签管理、标签事务和 SQLite 不是崩溃原因；
- `WindowManager.removeViewImmediate()` 不是本次栈顶原因；
- 单纯延长停止前等待时间只能降低概率，不能作为正式修复；
- 核心缺陷是 Editor 延迟任务未与 close/shutdown 生命周期绑定，同时 `displayMetrics()` 的 fallback 不具备空值防护。

上传的成功 JSON 为 `probeVersion=2`，其全部功能断言、隔离数据库关闭和运行锁释放均已完成；系统是在结果生成后被遗留延迟任务杀死。

## v3 生命周期修正

v3 不再停止并重新加载正式实例，而改为：

```text
正式实例 hide
→ 隔离实例启动并执行探测
→ 隔离实例 hide
→ 等待
→ 隔离实例 stop
→ 正式实例 show
```

该修正移除了正式实例 stop/restart 风险，但没有修复 Editor 自身的延迟 IME 竞态。

## v4 Editor 运行时防护

047 v4 在加载隔离 `ch_10_editor.js` 时应用仅限探测进程的确定性修正：

1. `displayMetrics()` 先检查 `windowManager` 和 `appContext`，服务已清空时返回空 `DisplayMetrics`，不再解引用 null；
2. 180 ms 初始 IME 测量捕获 `imePollGeneration`；
3. 回调执行前检查 generation、`ready`、`state.attached`、`panelRoot`、`appContext` 和 `windowManager`；
4. `clearViews()` 调用 `stopEditorImePolling()` 后 generation 变化，旧回调自动失效；
5. 结果新增：

```text
editorShutdownRaceGuardApplied=true
probeVersion=4
```

该运行时修正仅保护 047 隔离探测。正式业务模块 `src/ch_10_editor.js` 仍需应用同等核心修复并更新模块清单，不能把探测热补丁当作最终产品修复。
