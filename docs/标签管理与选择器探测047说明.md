探测版本：`v3`（安全生命周期版）。

要求模块集：

```text
moduleSetVersion=20260723.04
Editor=10
Filter=14
Settings=7
Repository=8

```

## 验证范围

1. 设置页采用 `reference_settings_v2`；
2. 标签名称、颜色预览、关联数量和拖动手柄正常；
3. 标签排序结果持久化到 `manual_order`；
4. 删除操作需要二次确认；
5. 删除标签只解除关联，不删除剪贴板记录；
6. 编辑页标签选择器采用 `reference_tag_selector_v1`；
7. 标签选择器取消或系统返回不写入 `clipboard_item_tags`；
8. 点击标签选择器“完成”后，仍需点击正文编辑页“保存”才提交关联；
9. 编辑已有记录与新增记录均支持多标签；
10. Filter Root 卡片使用标签自定义颜色，并继续显示两个标签和 `+N`；
11. Settings 返回、Editor 两层返回和唯一 Filter Root 保持正常；
12. 数据库 schema v2、Navigation v3、正式实例状态和隔离运行锁保持正常。

## v2 新记录修正

Editor 的 `openNew(options)` 不接收 `text` 作为初始正文。v1 的新增记录测试虽然传入了 `text`，实际编辑框仍为空，会导致新增记录保存断言误报。

加载器改为：

```text
openNew({ requestKeyboard:false })
→ Editor.setInputText("ClipHub 新记录标签保存测试")
→ 打开标签选择器
→ 完成标签草稿
→ 保存新记录
```

最终结果额外要求：

```text
newEditorTextSet=true
newEditorSave=true
newItemId>0
newItemTags=[tagC,tagA]
```

该修正只影响探测脚本，不修改 Editor v10 或业务模块。

## v3 系统重启修正

047 v2 会执行以下高风险链路：

```text
停止正式实例
→ 加载并启动隔离实例
→ 关闭隔离实例
→ 在同一个 system_server 会话内重新加载 15 个模块
→ 重新启动正式实例
```

在 Android 14、ShortX、`system_server`、uid=1000 环境下，真机出现正式 UI 消失后设备崩溃重启。当前尚无对应崩溃日志，因此不能把具体异常归因到单一模块；已确认需要先移除探测脚本中的强制正式实例 stop/restart 链路。

v3 改为：

```text
通过正式控制端点执行 hide
→ 正式后台实例、SQLite、剪贴板监听和运行锁继续保留
→ 在 ClipHubProbe047 隔离目录启动探测实例
→ 隔离实例先 hide
→ 等待 350 ms
→ stop 隔离实例并关闭隔离数据库
→ 通过正式控制端点执行 show
```

因此，v3 不再：

- 停止正式实例；
- 释放正式实例运行锁；
- 删除正式控制端点；
- 在探测末尾重新 `eval` 15 个正式模块；
- 在同一个探测会话内重新初始化正式数据库和广播接收器。

## 崩溃检查点

v3 会在结果文件中持续覆盖写入 `lastCheckpoint`。即使设备再次重启，也应保留最后一个已完成阶段：

```text
before_formal_hide
after_formal_hide
before_isolated_start
after_isolated_start
before_isolated_hide
after_isolated_hide
before_isolated_stop
after_isolated_stop
before_formal_restore
after_formal_restore
```

结果文件目录：

```text
shortx.getShortXDir()/ClipHub/probes/
```

若再次发生设备重启，重启后读取最新的：

```text
cliphub_tag_management_probe_047_*.json
```

重点提供其中的：

```text
probeVersion
lastCheckpoint
error
formalControl
hide
stop
formalRestore
```
