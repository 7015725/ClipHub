# ClipHub 编辑器关闭竞态修复与探测 048

## 已确认根因

047 v2 完成后约 110 ms，`system_server` 主线程执行 Editor 遗留的 180 ms IME 测量任务。此时 `Editor.shutdown()` 已将 `appContext`、`windowManager` 和 `mainHandler` 清空，遗留任务进入 `displayMetrics()` 后调用：

```text
appContext.getResources()
```

触发 Rhino `TypeError`，异常逃逸到 `system_server` 主线程，导致设备重启。

047 v4 使用探测级热修复后通过，证明根因与防护方向成立。

## 正式修复

正式修复已落地：

```text
Editor 10 → Editor 11
moduleSetVersion 20260723.04 → 20260723.05
commit=fe7809e73e6dfe6e44ce164a1b2b621b64055d4d
```

保持：

```text
entryVersion=4
databaseSchemaVersion=2
Navigation=3
15 模块结构
ClipHub.js 唯一入口
```

Editor v11 修复范围：

1. 所有匿名延迟回调统一经过 `postEditorDelayed()`；
2. 延迟回调绑定 `imePollGeneration`；
3. 执行前检查 `ready`、`state.attached`、`panelRoot`、`appContext` 和 `windowManager`；
4. 回调内部异常只记录到 Editor 状态，不允许逃逸到 `system_server` 主线程；
5. `displayMetrics()` 对空 `windowManager` 和空 `appContext` 安全退出；
6. `shutdown()` 先设置 `ready=false` 并停止 IME polling，再移除 UI 和清空上下文；
7. `runOnMainSync()` 在 `mainHandler` 已清空时明确失败；
8. 增加延迟任务发布、执行、取消、错误、待执行和 shutdown 后尝试计数。

