# ClipHub 编辑器关闭竞态修复与探测 048

## 已确认根因

047 v2 完成后约 110 ms，`system_server` 主线程执行 Editor 遗留的 180 ms IME 测量任务。此时 `Editor.shutdown()` 已将 `appContext`、`windowManager` 和 `mainHandler` 清空，遗留任务进入 `displayMetrics()` 后调用：

```text
appContext.getResources()
```

触发 Rhino `TypeError`，异常逃逸到 `system_server` 主线程，导致设备重启。

047 v4 使用探测级热修复后通过，证明根因与防护方向成立。

## 正式修复目标

正式修复升级：

```text
Editor 10 → Editor 11
moduleSetVersion 20260723.04 → 20260723.05
```

保持：

```text
entryVersion=4
databaseSchemaVersion=2
Navigation=3
15 模块结构
ClipHub.js 唯一入口
```

Editor v11 修复范围：

1. 所有匿名延迟回调统一经过 `postEditorDelayed()`；
2. 延迟回调绑定 `imePollGeneration`；
3. 执行前检查 `ready`、`state.attached`、`panelRoot`、`appContext` 和 `windowManager`；
4. 回调内部异常只记录到 Editor 状态，不允许逃逸到 `system_server` 主线程；
5. `displayMetrics()` 对空 `windowManager` 和空 `appContext` 安全退出；
6. `shutdown()` 先设置 `ready=false` 并停止 IME polling，再移除 UI 和清空上下文；
7. `runOnMainSync()` 在 `mainHandler` 已清空时明确失败；
8. 增加延迟任务发布、执行、取消、错误、待执行和 shutdown 后尝试计数。

## 应用正式修复

仓库已提供失败关闭的补丁工具：

```text
tools/apply_editor_shutdown_race_fix.py
tools/apply_editor_shutdown_race_fix.sh
```

Termux 执行：

```sh
cd "$HOME/ClipHub"
git fetch origin --prune
git switch agent/initialize-project-skeleton
git pull --ff-only origin agent/initialize-project-skeleton
chmod +x tools/apply_editor_shutdown_race_fix.sh
tools/apply_editor_shutdown_race_fix.sh --commit
```

执行器只修改：

```text
src/ch_10_editor.js
module-manifest.json
```

以下任一情况都会中止，不会继续写入：

- 当前分支不是 `agent/initialize-project-skeleton`；
- 两个目标文件已有本地修改或暂存修改；
- Editor v10 固定标记不匹配；
- 旧 Editor blob SHA 不匹配；
- 模块集不是 `20260723.04` 或已完成的 `20260723.05`；
- 修复后仍存在超过一个直接 `mainHandler.postDelayed`。



